Requisitos y Documentación

Este módulo explora los conceptos fundamentales sobre requisitos de software, sus tipos, técnicas de elicitación y buenas prácticas para su documentación.

Conceptos Fundamentales de Requisitos

Los requisitos de software son descripciones de los servicios que debe proporcionar un sistema y las restricciones bajo las cuales debe operar. Representan las necesidades y restricciones establecidas por los clientes y usuarios que ayudan a resolver un problema específico.

Los requisitos son fundamentales porque:

  • Definen qué debe hacer el sistema (y qué no debe hacer)
  • Establecen un contrato entre cliente y desarrollador
  • Sirven como base para la estimación de costos y cronogramas
  • Proporcionan la base para la validación y verificación
  • Facilitan la gestión del alcance del proyecto

Requisitos, Desarrollo y Validación

¿Sabías que...?

Según estudios del Standish Group, aproximadamente el 45% de las características desarrolladas en software nunca se utilizan, y otro 19% se usa raramente. Esto demuestra la importancia de una adecuada elicitación y validación de requisitos.

Características de Buenos Requisitos

Para que los requisitos sean efectivos, deben cumplir con las siguientes características:

Completos

Deben incluir toda la información necesaria sin omitir detalles importantes. Todo lo que el sistema debe hacer debe estar incluido.

Consistentes

No deben contradecirse entre sí ni con otros documentos del proyecto. No debe haber conflictos entre requisitos.

Verificables

Debe ser posible determinar si el requisito se ha implementado correctamente mediante pruebas o inspecciones.

No ambiguos

Deben tener una única interpretación. Todas las partes interesadas deben entender lo mismo al leerlos.

Priorizados

Deben tener una prioridad asignada para identificar los más críticos y poder gestionar adecuadamente el alcance.

Rastreables

Debe ser posible seguir el origen de cada requisito y su relación con otros elementos del sistema.

Tipos de Requisitos

Requisitos Funcionales

Los requisitos funcionales describen lo que el sistema debe hacer, sus funciones específicas o servicios que debe proporcionar.

Características:
  • Describen el comportamiento del sistema
  • Especifican entradas, salidas y procesamiento
  • Se expresan normalmente con verbos de acción
Ejemplos:
  • "El sistema debe permitir a los usuarios crear una nueva cuenta con correo electrónico y contraseña"
  • "El sistema debe enviar un correo de confirmación cuando un pedido es procesado"
  • "El sistema debe calcular automáticamente el total de una factura incluyendo impuestos"

Requisitos No Funcionales

Los requisitos no funcionales especifican criterios para evaluar el funcionamiento del sistema, en lugar de comportamientos específicos. Definen restricciones o atributos de calidad.

Categorías principales:
  • Rendimiento: Tiempo de respuesta, capacidad, uso de recursos
  • Seguridad: Autenticación, autorización, encriptación
  • Usabilidad: Facilidad de uso, accesibilidad, experiencia de usuario
  • Fiabilidad: Disponibilidad, tolerancia a fallos, recuperación
  • Mantenibilidad: Modularidad, capacidad de prueba, extensibilidad
  • Portabilidad: Adaptabilidad a diferentes entornos
Ejemplos:
  • "El tiempo de respuesta del sistema no debe exceder 2 segundos para cualquier operación"
  • "El sistema debe estar disponible el 99.9% del tiempo"
  • "La aplicación debe funcionar en navegadores Chrome, Firefox y Safari"

Otros Tipos de Requisitos

Requisitos de Dominio

Derivan del dominio de aplicación del sistema y reflejan características de ese dominio.

Ejemplo: "En una aplicación bancaria, el sistema debe cumplir con la normativa de protección de datos financieros"

Requisitos de Usuario

Describen los requisitos en lenguaje natural, desde la perspectiva del usuario.

Ejemplo: "Los usuarios deben poder filtrar productos por categoría y precio"

Requisitos de Sistema

Descripciones más detalladas y técnicas de las funciones que el sistema debe proporcionar.

Ejemplo: "El módulo de autenticación debe utilizar el protocolo OAuth 2.0"

Requisitos de Proceso

Restricciones en el proceso de desarrollo del software.

Ejemplo: "El código debe pasar todas las pruebas unitarias antes de ser integrado"

Técnicas de Elicitación de Requisitos

La elicitación de requisitos es el proceso de descubrir, extraer y recopilar las necesidades de los stakeholders. Existen diversas técnicas que pueden clasificarse en dos categorías principales:

Técnicas Centradas en la Documentación

Estas técnicas se basan en la revisión de documentos existentes y la generación de documentación.

Descripción: Revisión de manuales, procesos, reportes y documentación existente relacionada con el sistema actual o sistemas similares.

Beneficios:

  • Comprensión de procesos actuales
  • Identificación de reglas de negocio
  • Menor tiempo de interrupción para los usuarios

Descripción: Recolección estructurada de información mediante preguntas predefinidas dirigidas a múltiples stakeholders.

Beneficios:

  • Alcance a gran número de interesados
  • Estadísticas y datos cuantificables
  • Facilidad para comparar respuestas

Ejemplo: Encuesta a usuarios finales sobre funcionalidades más importantes en un sistema existente.

Descripción: Empleo de estructuras estandarizadas y formatos predefinidos para capturar requisitos específicos.

Beneficios:

  • Consistencia en la documentación
  • Menor probabilidad de olvidar aspectos importantes
  • Facilita la reutilización de requisitos comunes

Ejemplo: Plantilla de historia de usuario, plantilla de caso de uso, o plantillas de requisitos no funcionales.

Descripción: Creación de representaciones gráficas de los procesos de negocio para entender el flujo de información y actividades.

Beneficios:

  • Visualización clara de procesos complejos
  • Identificación de ineficiencias y oportunidades de mejora
  • Facilita la comunicación entre stakeholders técnicos y no técnicos

Notaciones comunes: BPMN (Business Process Model and Notation), Diagramas de flujo, Diagramas de actividad UML.

Técnicas de Observación

Estas técnicas implican la interacción directa con los usuarios y la observación de sus actividades.

Descripción: Conversación directa con stakeholders para obtener información detallada sobre sus necesidades y expectativas.

Tipos:

  • Estructuradas: Preguntas predefinidas en un orden específico
  • Semi-estructuradas: Preguntas planificadas pero con espacio para exploración
  • No estructuradas: Conversación abierta sobre un tema específico

Beneficios: Información detallada, aclaración inmediata de dudas, captura de requisitos implícitos.

Descripción: Observación del entorno de trabajo y las actividades de los usuarios en tiempo real.

Variantes:

  • Observación pasiva: El analista observa sin intervenir
  • Observación participativa: El analista participa en las actividades

Beneficios: Descubrimiento de requisitos no expresados, comprensión del contexto real de uso, identificación de problemas prácticos.

Descripción: Sesiones estructuradas donde usuarios, analistas y desarrolladores trabajan conjuntamente para definir requisitos.

Características:

  • Reuniones altamente focalizadas
  • Participación de múltiples stakeholders
  • Uso de facilitadores para guiar la sesión
  • Documentación en tiempo real

Beneficios: Consenso entre diferentes stakeholders, reducción del tiempo de elicitación, identificación temprana de conflictos.

Descripción: Desarrollo de versiones preliminares del sistema para obtener feedback temprano de los usuarios.

Tipos:

  • Prototipos de baja fidelidad: Bocetos, wireframes
  • Prototipos de alta fidelidad: Diseños detallados, interactivos
  • Prototipos desechables: Solo para obtener feedback, luego se descarta
  • Prototipos evolutivos: Se convierten gradualmente en el producto final

Beneficios: Visualización temprana de soluciones, validación de conceptos, descubrimiento de requisitos ocultos.

Buenas Prácticas de Documentación (Norma APA)

La documentación adecuada de los requisitos es esencial para su gestión efectiva. La Norma APA proporciona lineamientos útiles para la estructuración y formato de documentos técnicos:

Principios Generales de Documentación APA

  • Claridad y precisión: Usar lenguaje claro y directo
  • Organización jerárquica: Estructurar el contenido con encabezados y subencabezados
  • Citación adecuada: Reconocer fuentes de información
  • Formato consistente: Mantener coherencia en estilo y presentación

Aplicación en Documentos de Requisitos

  • Referencias a documentos técnicos: Citar correctamente estándares, normativas o documentos previos
  • Estructura organizada: Facilita la navegación y comprensión del documento
  • Numeración de requisitos: Sistema jerárquico para identificación y trazabilidad
  • Uso de tablas y figuras: Complementar texto con elementos visuales numerados y referenciados

Documento de Especificación de Requisitos (SRS)

Un documento SRS bien estructurado según principios similares a APA debería incluir:

  1. Introducción: Propósito, alcance, definiciones, referencias
  2. Descripción general: Perspectiva del producto, funciones, características de usuarios, restricciones
  3. Requisitos específicos: Detalle de todos los requisitos organizados jerárquicamente
  4. Apéndices: Diagramas, modelos, información complementaria
  5. Índices: Facilitan la navegación en documentos extensos

Facilitación Gráfica

La facilitación gráfica es una técnica que utiliza elementos visuales para capturar, organizar y comunicar ideas de manera efectiva. Es especialmente útil en la elicitación y documentación de requisitos porque:

  • Facilita la comprensión de conceptos complejos
  • Mejora la retención y el recuerdo de la información
  • Promueve la participación y colaboración
  • Ayuda a identificar relaciones y patrones
  • Reduce ambigüedades y malas interpretaciones

Herramientas de Facilitación Gráfica para Requisitos

Diagramas de Casos de Uso

Representan la interacción entre los usuarios (actores) y el sistema, mostrando las funcionalidades desde la perspectiva del usuario.

Útiles para: Capturar requisitos funcionales de alto nivel y delimitar el alcance del sistema.

Mapas Mentales

Organizan visualmente información alrededor de un concepto central, mostrando relaciones jerárquicas y asociativas.

Útiles para: Realizar tormenta de ideas, categorizar requisitos y mostrar relaciones entre conceptos.

Wireframes y Mockups

Representaciones visuales de la interfaz de usuario en diferentes niveles de fidelidad, desde esquemas básicos hasta diseños detallados.

Útiles para: Visualizar requisitos de interfaz, validar flujos de usuario y obtener feedback temprano.

Diagramas de Flujo

Muestran paso a paso el flujo de un proceso o algoritmo, utilizando símbolos estandarizados para representar diferentes tipos de acciones.

Útiles para: Documentar procesos de negocio, algoritmos y lógica de decisión.

Tableros de Historias de Usuario

Organizan visualmente las historias de usuario, permitiendo priorización, agrupación y seguimiento del estado.

Útiles para: Gestionar requisitos en metodologías ágiles, planificar sprints y visualizar el progreso.

Diagramas de Entidad-Relación

Representan las entidades de un sistema y cómo se relacionan entre sí, modelando la estructura de datos.

Útiles para: Diseñar bases de datos, modelar dominios complejos y visualizar requisitos de datos.

Caso Práctico: Historias de Usuario

Las historias de usuario son una técnica popular en metodologías ágiles para capturar requisitos desde la perspectiva del usuario. Veamos cómo se pueden documentar y utilizar:

Estructura de una Historia de Usuario

Formato básico:

Como [tipo de usuario]
Quiero [objetivo]
Para [beneficio/valor]

Componentes adicionales:

  • Criterios de Aceptación: Condiciones que deben cumplirse para considerar la historia como completada
  • Estimación: Esfuerzo relativo requerido (puntos de historia)
  • Prioridad: Importancia relativa para el negocio
  • Notas Técnicas: Consideraciones adicionales para el equipo de desarrollo

Ejemplo de Historia de Usuario

HU-12 Alta prioridad
Buscar productos por categoría

Como cliente de la tienda online
Quiero poder buscar productos por categoría
Para encontrar más fácilmente lo que estoy buscando

Criterios de aceptación:

  • Debe mostrarse un menú desplegable con todas las categorías disponibles
  • Al seleccionar una categoría, sólo se muestran productos de esa categoría
  • Los resultados deben mostrarse paginados, 20 productos por página
  • Debe haber un contador que muestre el total de productos en esa categoría
Estimación: 5 puntos Sprint: 2

  • Tipos de requisitos: Funcionales, no funcionales, de usuario, de sistema, de dominio y de proceso
